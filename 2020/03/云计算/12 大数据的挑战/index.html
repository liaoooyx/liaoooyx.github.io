大数据

发表于 2020-03-16 更新于 2020-05-28 分类于 云计算 ， 课件
本文字数： 796 阅读时长 ≈ 2 分钟

数据的类型

关系型数据：表、事务、遗产

文本数据：Web

半结构化数据：XML

图形数据：社交网络、语义网络

流数据：只能被扫描一次

大数据的5Vs

Volume 体积

Velocity 速度

Variety 多样性

Veracity 真实性

Value 价值

专用场景

  • 索引，搜索，查询
    • 基于关键字的搜索
    • 模式匹配（XML / RDF）
    • 深度学习
  • 知识发现
    • 数据挖掘
    • 统计建模
  • 聚合与统计
  • 建模与预测

挑战

  • 大量数据
    • 数据产生和类型的速度正在迅速提高
  • 数据描述
    • 用于描述内容的元数据和语义
  • 搜索
    • 启用语义的搜索引擎
  • 储存和保留
    • 重建比存储更容易
    • 有用数据/噪音数据的比率
    • 公司通常存储所有内容
  • 监管数据收集
    • 信息和知识
    • 质量控制

可能的解决方案：并行化

并行化的挑战

  • 调度

    • 我们如何将工作分配给单个workers？

  • 可用性

    • 如果工作单位多于可用的workers该怎么办？

  • 依赖关系

    • 如果worker需要共享部分结果怎么办？
    • 我们如何知道所有worker何时完成？

  • 容错

    • 如果worker在执行过程中故障怎么办？

  • 通信

    • workers之间交换状态，访问共享资源（即数据）
需要一种同步机制

编程模型

  • 共享内存：pthreads

  • 消息传递：MPI

设计模式

  • Master-slaves 主从

  • 生产者消费者

并行化的挑战2

以下情况会使【并发操作】变得困难

  • 数据中心扩展
  • 故障和中断
  • 多种交互服务
  • 调试

负担转移

  • 开发人员指定需要执行的计算
  • 处理实际执行的执行框架（在运行时）
MapReduce、DFS

略

案例学习：谷歌文件系统 GFS

总体结构

GFS假设

  • 使用普通的商品硬件
  • 高组件故障率
    • 廉价的商品组件会一直故障（使用心跳检测）
  • 数量不多的大型文件
    • 千兆字节的文件很常见
  • 文件是一次写入的，大部分附加到（同时）

GFS设计

  • 文件作为【chunks】被储存
    • 固定大小（64MB）
  • 通过复制来保证可靠性
    • 每个【chunk】会被复制到超过3个【chunkserver】
  • 单个【master】来协调访问
    • 简单的中心化管理
  • 没有数据缓存
    • 因为巨大的数据集，因此缓存的作用很低
  • 简化接口
使用场景 大数据挑战 同步机制 GFS
