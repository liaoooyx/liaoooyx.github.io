服务质量QoS 和 服务级别协议SLAs

QoS是描述服务质量的属性

SLA是保证服务质量的实施方法之一，他是供应商和使用方双方的协定

服务质量：QoS：一个（系统）元素（如应用程序、作业、服务）具有一定程度的保证可以满足其服务要求的能力。

  • 比如使用Google搜索，我搜索的响应在1s内，那么谷歌搜索的QoS是有保障的，或者说我得到了服务质量

云计算使用者所强调的核心缺陷，包括：

  • 默认提供“尽最大努力”的服务
    • QoS使得服务提供商尽最大的努力提供服务，但不做任何保证。
  • QoS => 随着时间的推移，保证资源使用
  • 性能、安全、信任、成本

所以引出了服务级别协议，来解决了缺陷：

  • 定义了商业关系
  • 专注于服务质量感知中间件/系统的开发

服务提供商和SLAs

服务提供商必须保证SLA，但对采用SLA持谨慎态度。因为服务提供商不得不面对一些商业风险，比如由于【故障】、【拒绝服务攻击】、【过载】等导致SLA违规和惩罚。

服务供应商是否应该提供SLA，考虑：

  • 要提供的QoS级别？
  • 是否有足够的资源来执行服务？
  • 是否有容错能力？
  • 要采取的行动？

问题：（云）管理域

分布式系统中的【QoS预留】要求资源状态为：已设置、已存储、已维护

状态信息：是什么？在哪里？什么时候？有多少？

通用问题：

  • 缩放scaling
  • 会计/收费

  • 交付内容的管理包括：协议双方的角色，权利和义务

Amazon EC2 SLAs 所包含的内容有：

  • 服务承诺和定义
  • 服务积分的获得和使用范围
  • Amazon EC2 SLA排除项

服务级别协议 SLA

【服务级别协议】是在两个组织（服务提供商和客户）之间的业务关系中存在的对期望和义务的明确声明。

  • Purpose：描述协议的预期目标
  • Parties：参与协议或与协议有利益或目的的各方
  • Scope：协议涵盖的区域
  • SLObjectives：可以保证的度量指标以及要维持的指标水平
  • State：描述SLA的当前状态
  • Exclusions：什么不包括，例如，工作监控，保留或调整

描述
Purpose 保证运行云计算密集型应用程序
确保满足用户要求
Parties 用户，经纪人，资源
Scope 计算服务
Service Level Objectives (SLO) 资源可用性
属性：CPU数量，CPU类型，CPU速度，RAM，存储，操作系统和版本
Service Level Indicators (SLI) 资源可用性和每个属性的值（详见WS-Agreement小节图）
Exclusions 指定所提供的服务等级，例如 服务监控/预订/适应
Administration 通过重新协商/适应资源来实现SLA的目标

SLA的内容

在一定时间内必须达到指定数量和质量的资源才能达到预期的性能

  • 在从资源【所有者】到【请求者】的定义时间间隔内，委派特定资源功能

数据的类型

关系型数据：表、事务、遗产

文本数据：Web

半结构化数据：XML

图形数据：社交网络、语义网络

流数据：只能被扫描一次

大数据的5Vs

Volume 体积

Velocity 速度

Variety 多样性

Veracity 真实性

Value 价值

专用场景

  • 索引，搜索，查询
    • 基于关键字的搜索
    • 模式匹配（XML / RDF）
    • 深度学习
  • 知识发现
    • 数据挖掘
    • 统计建模
  • 聚合与统计
  • 建模与预测

挑战

  • 大量数据
    • 数据产生和类型的速度正在迅速提高
  • 数据描述
    • 用于描述内容的元数据和语义
  • 搜索
    • 启用语义的搜索引擎
  • 储存和保留
    • 重建比存储更容易
    • 有用数据/噪音数据的比率
    • 公司通常存储所有内容
  • 监管数据收集
    • 信息和知识
    • 质量控制

可能的解决方案：并行化

并行化的挑战

  • 调度

    • 我们如何将工作分配给单个workers？

  • 可用性

    • 如果工作单位多于可用的workers该怎么办？

  • 依赖关系

    • 如果worker需要共享部分结果怎么办？
    • 我们如何知道所有worker何时完成？

  • 容错

    • 如果worker在执行过程中故障怎么办？

  • 通信

    • workers之间交换状态，访问共享资源（即数据）
需要一种同步机制

编程模型

  • 共享内存：pthreads

  • 消息传递：MPI

设计模式

  • Master-slaves 主从

  • 生产者消费者

并行化的挑战2

以下情况会使【并发操作】变得困难

  • 数据中心扩展
  • 故障和中断
  • 多种交互服务
  • 调试

负担转移

  • 开发人员指定需要执行的计算
  • 处理实际执行的执行框架（在运行时）
MapReduce、DFS

略

案例学习：谷歌文件系统 GFS

总体结构

GFS假设

  • 使用普通的商品硬件
  • 高组件故障率
    • 廉价的商品组件会一直故障（使用心跳检测）
  • 数量不多的大型文件
    • 千兆字节的文件很常见
  • 文件是一次写入的，大部分附加到（同时）

GFS设计

  • 文件作为【chunks】被储存
    • 固定大小（64MB）
  • 通过复制来保证可靠性
    • 每个【chunk】会被复制到超过3个【chunkserver】
  • 单个【master】来协调访问
    • 简单的中心化管理
  • 没有数据缓存
    • 因为巨大的数据集，因此缓存的作用很低
  • 简化接口
